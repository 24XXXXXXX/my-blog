很多人在第一次把项目放到服务器上时，都会遇到同样的问题：

- 前端和后端要不要分别打包？
- 文件到底怎么传到服务器，`scp` 还是 `GitHub`？
- 本地代码更新后，怎么同步到服务器？
- `Nginx` 到底要不要用？它在部署里负责什么？

这篇文章会把这些问题一次讲清楚，并给出一套适合大多数个人项目和中小项目的部署思路。

<!--more-->

## 一、先说结论

如果你的项目是典型的前后端分离项目，最常见、也最合理的部署方式是：

1. **前端和后端分别部署**
2. **前端先构建出静态文件，再放到服务器**
3. **后端打包成可运行产物，再在服务器上启动**
4. **Nginx 负责对外提供访问入口**
5. **后续更新优先通过 Git 或 CI/CD 同步，不建议长期手工传文件**

可以先记住下面这句话：

> `scp` 适合临时上传，`Git pull` 适合简单更新，`CI/CD` 适合长期维护，`Nginx` 负责对外访问和反向代理。

---

## 二、前端和后端要不要分别打包

通常要。

### 2.1 前端项目

前端项目例如：

- `Vue`
- `React`
- `Vite`
- `Next.js` 的静态导出

如果是普通前端项目，最终部署到服务器上的一般不是源码，而是构建后的静态文件，例如：

```text
dist/
build/
out/
```

也就是说：

- 你本地写的是源码
- 真正放到服务器上给用户访问的是打包结果

### 2.2 后端项目

后端项目例如：

- `Spring Boot`
- `Node.js / Express / NestJS`
- `Django`
- `Flask`

后端部署到服务器上的一般是可运行产物，比如：

- `jar` 包
- `war` 包
- `node` 项目源码加依赖
- `Python` 项目源码加虚拟环境
- `Docker` 镜像

例如：

- Java 后端常见是 `target/app.jar`
- Node 后端常见是整份项目代码，然后在服务器 `npm install` 后启动

所以，前后端在大多数情况下确实是分开部署的。

## 三、文件怎么发到服务器

常见有 4 种方式：

1. `scp`
2. `rsync`
3. `GitHub / Gitee + git pull`
4. `CI/CD`

### 3.1 `scp`：最直接，但不适合长期手工维护

`scp` 适合：

- 你刚学部署
- 项目体积不大
- 临时上传一个打包文件
- 快速验证线上环境

例如上传前端打包结果：

```bash
scp -r dist/* admin@服务器IP:/var/www/myapp/
```

上传 Java 后端包：

```bash
scp target/myapp.jar admin@服务器IP:/opt/myapp/backend/
```

优点：

- 简单
- 立即可用
- 不需要在服务器上拉代码

缺点：

- 每次都手工传文件，容易漏
- 不方便做版本回滚
- 不适合多人协作
- 对大量文件更新效率一般

### 3.2 `rsync`：比 `scp` 更适合反复同步

如果你已经进入“经常更新”的阶段，比起 `scp`，更推荐 `rsync`。

例如同步前端静态资源：

```bash
rsync -avz --delete dist/ admin@服务器IP:/var/www/myapp/
```

参数说明：

- `-a`：保留文件属性
- `-v`：显示详细过程
- `-z`：压缩传输
- `--delete`：删除目标目录中本地已不存在的文件

优点：

- 增量同步，速度更快
- 适合频繁更新
- 比 `scp` 更适合部署场景

缺点：

- 仍然偏手工
- 只是“同步文件”，不负责版本管理和自动重启

### 3.3 `GitHub + git pull`：个人项目最常见

这是很多个人项目和小团队最常见的做法：

1. 本地写代码
2. 提交到 GitHub / Gitee
3. 服务器进入项目目录执行 `git pull`
4. 重新构建或重启服务

例如：

```bash
cd /opt/myapp
git pull origin main
```

优点：

- 代码版本清晰
- 能配合分支管理
- 更新路径固定
- 比手工传文件更规范

缺点：

- 服务器要有 Git 环境
- 如果你直接在生产目录里 `git pull`，更新过程不够优雅
- 如果服务器上也改过文件，容易冲突

### 3.4 `CI/CD`：长期最推荐

更规范的方式通常是：

1. 本地开发
2. 推送到 GitHub
3. GitHub Actions / Jenkins / GitLab CI 自动构建
4. 自动上传产物或镜像到服务器
5. 自动重启服务

优点：

- 自动化程度高
- 不容易手误
- 可重复、可追踪
- 适合后续长期维护

缺点：

- 初次配置成本更高
- 对初学者来说略复杂

---

## 四、到底选哪种方式

可以按下面这个标准判断。

| 场景 | 推荐方式 |
|------|----------|
| 只是临时上线一次 | `scp` |
| 经常同步静态文件 | `rsync` |
| 个人项目、轻量更新 | `GitHub + git pull` |
| 想长期稳定维护 | `CI/CD` |

如果你现在还在入门阶段，我更建议你用下面这条路线：

1. 代码放在 GitHub
2. 服务器上用 `git pull` 更新
3. 前端重新构建或直接上传构建产物
4. 后端重启服务
5. 用 Nginx 做反向代理和静态资源服务

这条路线比全手工 `scp` 更规范，但又没有直接上 CI/CD 那么复杂。

## 五、本地更新后，怎么同步给服务器

这是最关键的问题。

正确理解是：

- **本地开发机是写代码的地方**
- **GitHub 是代码仓库**
- **服务器是运行环境**

不建议把服务器当成主要开发环境，也不建议在线上服务器里直接改正式代码。

### 5.1 最常见的同步流程

推荐流程：

1. 本地修改代码
2. 本地测试通过
3. 提交到 Git
4. 推送到 GitHub
5. 服务器拉取最新代码或接收最新构建产物
6. 重启服务

也就是：

```text
本地开发 -> push 到 GitHub -> 服务器 pull / 部署 -> 重启服务
```

### 5.2 你的理解基本是对的

你提到的这句：

> 更新后上传 github, 然后服务器拉取覆盖旧项目

这是非常常见、也非常合理的做法。

但“覆盖旧项目”更准确的说法应该是：

- 拉取最新版本
- 重新构建
- 替换当前运行版本

而不是手工把旧文件乱覆盖一遍。

### 5.3 不推荐的方式

不推荐长期这样做：

- 本地改一点
- 手工 `scp` 几个文件到服务器
- 服务器里再手工改一点

原因很简单：

- 本地和线上很快就不同步
- 你会忘记线上改过什么
- 出问题时很难回滚

## 六、推荐的实际部署流程

下面给一套适合大多数前后端分离项目的实用流程。

## 方案 A：前端静态文件 + 后端服务

适合：

- Vue
- React
- Vite
- Spring Boot
- Node.js API 服务

### 6.1 前端部署流程

本地构建前端：

```bash
npm install
npm run build
```

构建后会得到：

```text
dist/
```

然后把 `dist` 上传到服务器，例如：

```bash
rsync -avz --delete dist/ admin@服务器IP:/var/www/myapp/
```

或者：

```bash
scp -r dist/* admin@服务器IP:/var/www/myapp/
```

### 6.2 后端部署流程

#### Java 后端

本地打包：

```bash
mvn clean package -DskipTests
```

上传：

```bash
scp target/myapp.jar admin@服务器IP:/opt/myapp/backend/
```

服务器上运行：

```bash
cd /opt/myapp/backend
nohup java -jar myapp.jar > app.log 2>&1 &
```

#### Node.js 后端

方式一：服务器拉代码后构建：

```bash
cd /opt/myapp/backend
git pull origin main
npm install
npm run build
node dist/main.js
```

方式二：本地构建后上传产物。

但对 Node 项目来说，更常见的是：

- 代码推到 GitHub
- 服务器 `git pull`
- 服务器安装依赖并启动

## 七、前端和后端推荐放在哪里

在服务器上建议分开目录，别混在一起。

例如：

```text
/var/www/myapp/                 # 前端静态文件
/opt/myapp/backend/            # 后端程序
/opt/myapp/logs/               # 日志
/etc/nginx/sites-available/    # Nginx 配置
```

这是比较常见的做法：

- `/var/www` 适合静态网站文件
- `/opt` 适合应用程序
- `/etc/nginx` 放 Nginx 配置

---

## 八、后端服务不要只靠 `nohup`

很多人刚开始都会这样启动：

```bash
nohup java -jar myapp.jar > app.log 2>&1 &
```

能用，但不够稳。

更推荐使用：

- `systemd`
- 或者 Node 项目用 `pm2`

### 8.1 `systemd` 管理 Java 服务

创建服务文件：

```bash
sudo nano /etc/systemd/system/myapp.service
```

内容示例：

```ini
[Unit]
Description=My Java App
After=network.target

[Service]
User=www-data
WorkingDirectory=/opt/myapp/backend
ExecStart=/usr/bin/java -jar /opt/myapp/backend/myapp.jar
SuccessExitStatus=143
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

加载并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable myapp
sudo systemctl start myapp
```

查看状态：

```bash
sudo systemctl status myapp
```

重启服务：

```bash
sudo systemctl restart myapp
```

查看日志：

```bash
journalctl -u myapp -f
```

### 8.2 Node.js 后端用 `pm2`

安装：

```bash
npm install -g pm2
```

启动：

```bash
pm2 start dist/main.js --name myapp-api
```

查看：

```bash
pm2 list
pm2 logs myapp-api
```

重启：

```bash
pm2 restart myapp-api
```

开机自启：

```bash
pm2 startup
pm2 save
```

## 九、Nginx 在部署中有什么用

通常有，而且非常常用。

Nginx 在部署里的主要作用有 5 个：

1. **提供前端静态文件**
2. **反向代理后端服务**
3. **统一对外暴露 80 / 443 端口**
4. **处理 HTTPS 证书**
5. **把前端和后端整合到同一个域名下**

你可以把它理解成：

- 用户访问的是 `Nginx`
- Nginx 再决定把请求交给前端文件还是后端服务

例如：

- `https://example.com/` -> 前端页面
- `https://example.com/api/` -> 后端接口

---

## 十、什么时候必须用 Nginx

以下场景几乎都建议使用 Nginx：

- 前端是静态页面
- 后端跑在 `8080`、`3000`、`5000` 等端口
- 你希望外部只访问 `80` 或 `443`
- 你要配置域名
- 你要做 HTTPS
- 你要做跨域规避或统一网关入口

如果没有 Nginx，也不是绝对不能部署，但会遇到很多问题：

- 前端静态资源服务不够专业
- 后端直接暴露端口不美观
- 不方便配域名和证书
- 不方便做反向代理

所以，大多数正式部署场景都建议使用 Nginx。

## 十二、Nginx 配置文件在哪里

Ubuntu 常见路径：

```text
/etc/nginx/nginx.conf
/etc/nginx/sites-available/
/etc/nginx/sites-enabled/
```

通常做法是：

1. 在 `sites-available` 写站点配置
2. 用软链接链接到 `sites-enabled`
3. 测试配置
4. 重载 Nginx

例如：

```bash
sudo nano /etc/nginx/sites-available/myapp
sudo ln -s /etc/nginx/sites-available/myapp /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx
```

## 十三、Nginx 部署前端静态项目示例

假设前端打包后的文件放在：

```text
/var/www/myapp/
```

配置示例：

```nginx
server {
    listen 80;
    server_name your-domain.com;

    root /var/www/myapp;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

这个配置的关键点：

- `root /var/www/myapp;` 指定前端静态文件目录
- `index index.html;` 指定首页
- `try_files $uri $uri/ /index.html;` 让 SPA 路由正常刷新不 404

这对 `Vue Router`、`React Router` 这类前端路由非常重要。

---

## 十四、Nginx 反向代理后端接口示例

假设后端运行在：

```text
127.0.0.1:8080
```

配置示例：

```nginx
server {
    listen 80;
    server_name api.your-domain.com;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

这表示：

- 浏览器访问 `api.your-domain.com`
- 实际流量会转发到本机 `8080`

## 十五、前端和后端同域名部署示例

这是最常见的生产部署方式之一。

例如：

- 前端：`https://your-domain.com/`
- 后端：`https://your-domain.com/api/`

Nginx 配置：

```nginx
server {
    listen 80;
    server_name your-domain.com;

    root /var/www/myapp;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8080/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

这种方式的好处：

- 用户只记一个域名
- 前后端入口统一
- 更容易处理跨域问题

## 十七、推荐的最小可用部署方案

如果你现在刚开始，最简单但相对规范的方案可以这样定：

### 前端

- 本地 `npm run build`
- 用 `rsync` 上传 `dist`
- Nginx 提供静态资源

### 后端

- Java：本地打 `jar`，上传后 `systemd` 重启
- Node：推到 GitHub，服务器 `git pull` 后用 `pm2` 重启

### 网关

- 用 Nginx 对外统一暴露 `80/443`
- `/` 指向前端
- `/api/` 反向代理到后端

这是对初学者非常友好的一条路线。

## 十八、什么时候该从 `scp` 升级到 Git 或 CI/CD

如果你开始出现这些情况，就该升级部署方式了：

- 更新越来越频繁
- 有多人协作
- 线上回滚需求明显
- 每次部署步骤太多，容易漏
- 想减少手工操作

升级顺序建议：

1. `scp`
2. `rsync`
3. `Git pull`
4. `CI/CD`

你不需要一开始就把系统搞得特别复杂，但也不要长期停留在“纯手工复制文件”的状态。

## 十九、常见错误

### 1. 直接把源码目录整个扔进 `/var/www`

不推荐。

前端通常只需要上传构建后的静态文件，而不是整个源码仓库。

### 2. 线上直接改代码

不推荐。

这样很快就会导致：

- 本地代码和线上代码不一致
- Git 记录失真
- 回滚困难

### 3. 更新后忘记重启后端

后端代码更新后，通常需要重启服务，否则新代码不会生效。

### 4. 改了 Nginx 配置但没测试

正确顺序应该是：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

不要改完就直接重启。

### 5. 前端 SPA 刷新 404

通常是 Nginx 没有加：

```nginx
try_files $uri $uri/ /index.html;
```

## 二十、总结

部署项目时，你可以这样理解：

1. **前端和后端通常要分别部署**
2. **前端一般上传构建产物，不是直接上传源码**
3. **后端一般上传可运行产物，或者在服务器拉代码后构建**
4. **长期来看，推荐“本地开发 -> GitHub -> 服务器更新”**
5. **Nginx 在正式部署中非常常用，负责静态资源、反向代理、域名和 HTTPS**

如果你现在只想要一套最实用的答案，可以直接照下面这条做：

```text
前端：本地 build -> rsync 上传 dist -> Nginx 提供访问
后端：代码 push 到 GitHub -> 服务器 pull -> 重启服务
Nginx：把 / 指向前端，把 /api/ 转发给后端
```

这已经足够支撑大多数个人项目和课程项目上线。
