SSH 隧道转发可以把你本机已经可用的代理能力，通过加密的 SSH 连接“带到”远程服务器上使用。对于“服务器本身无法直接访问外网，但你的电脑可以正常代理上网”的场景，这是比直接暴露代理端口更安全、也更稳定的做法。

<!--more-->

## 一、什么是 SSH 隧道转发

本文讨论的核心命令是：

```bash
ssh -R 10808:127.0.0.1:10808 admin@服务器IP -N
```

这里的 `-R` 表示 **反向端口转发**：

- 远程服务器监听 `127.0.0.1:10808`
- 当服务器上的程序访问这个端口时，流量会通过 SSH 隧道回到你的本地电脑
- 然后再由本地电脑连接 `127.0.0.1:10808`
- 也就是连接到你本机的代理软件

换句话说，服务器上访问的是 `127.0.0.1:10808`，真正提供代理能力的却是你本地电脑。

---

## 二、适用场景

这种方式常见于以下场景：

- 云服务器本身无法访问 Google、GitHub、OpenAI 等外网服务
- 本地电脑已经安装了 Clash、V2Ray、v2rayN、sing-box 等代理软件
- 不希望把本地代理端口直接暴露到公网
- 希望服务器只临时借用本地代理能力

相比“直接开放一个公网代理端口”，SSH 隧道的优势很明确：

- 流量走现有 SSH 加密通道
- 不需要额外开放代理服务到公网
- 默认只在服务器本机 `127.0.0.1` 上监听，更安全
- 连接断开后端口自然失效，风险更低

## 三、操作步骤

### 步骤 1：本机代理允许局域网连接

在你的代理软件中，例如 Clash、Clash Verge、V2Ray、v2rayN：

- 开启“允许局域网连接”（`Allow LAN`）

这一步在很多环境下是有帮助的，尤其是以下情况：

- 你的 SSH 客户端运行在 WSL、Docker、虚拟机中
- 代理软件和 SSH 客户端不在同一个网络命名空间
- 你后续还想让局域网其他设备复用该代理

如果你的代理软件和执行 `ssh -R` 的终端就在同一台本机上，并且都直接使用 `127.0.0.1`，那么这一步有时不是硬性要求；但为了减少环境差异导致的问题，建议先开启。

你还需要确认本地代理端口确实可用。比如如果你的 HTTP 或 mixed 代理端口就是 `10808`，可以测试：

```bash
curl -I --proxy http://127.0.0.1:10808 https://www.google.com
```

如果你的 `10808` 是 SOCKS5 端口，则改用：

```bash
curl -I --proxy socks5h://127.0.0.1:10808 https://www.google.com
```

> 注意：很多代理工具里 `10808` 常常是 `SOCKS5` 端口，而不是 `HTTP` 端口。  
> 如果你用错协议，后面即使 SSH 隧道建立成功，服务器端依然会表现为“代理不可用”。

---

### 步骤 2：建立 SSH 隧道

在本地电脑新开一个终端执行：

```bash
# 本地执行：把本地代理端口转发到服务器的 10808 端口
ssh -R 10808:127.0.0.1:10808 admin@服务器IP -N
```

参数说明：

- `-R 10808:127.0.0.1:10808`：把服务器上的 `10808` 端口转发到本地的 `127.0.0.1:10808`
- `admin@服务器IP`：你的 SSH 登录账号和服务器地址
- `-N`：不执行远程命令，只建立端口转发

执行成功后：

- 保持这个终端不要关闭
- 只要 SSH 会话还在，服务器端的 `127.0.0.1:10808` 就会一直可用

如果你希望 SSH 在端口转发失败时立即退出，可以用更稳妥的写法：

```bash
ssh -o ExitOnForwardFailure=yes -R 10808:127.0.0.1:10808 admin@服务器IP -N
```

---

### 步骤 3：在服务器上配置代理

先明确一个关键点：

- 如果你本地 `10808` 是 `HTTP` 或 `mixed` 代理端口，那么服务器端可以直接设置 `http_proxy`
- 如果你本地 `10808` 是 `SOCKS5` 端口，那么应当设置 `ALL_PROXY=socks5h://...`

### 情况 A：服务器上使用 HTTP 代理

如果你的本地代理软件把 `10808` 提供为 `HTTP` 或 `mixed` 端口，可以在服务器上执行：

```bash
export http_proxy=http://127.0.0.1:10808
export https_proxy=http://127.0.0.1:10808
```

### 情况 B：服务器上使用 SOCKS5 代理

如果你的本地 `10808` 实际上是 SOCKS5 端口，更推荐这样写：

```bash
export ALL_PROXY=socks5h://127.0.0.1:10808
```

有些程序同时识别以下写法，也可以一起设置：

```bash
export all_proxy=socks5h://127.0.0.1:10808
export ALL_PROXY=socks5h://127.0.0.1:10808
```

---

### 步骤 4：测试

### HTTP 代理测试

如果你使用的是 HTTP 代理环境变量：

```bash
curl -I https://www.google.com
```

### SOCKS5 代理测试

如果你使用的是 SOCKS5，更稳妥的测试方式是：

```bash
curl -I --proxy socks5h://127.0.0.1:10808 https://www.google.com
```

如果返回 `HTTP/2 200`、`HTTP/1.1 200 OK`、`301`、`302` 等响应头，通常说明链路已经打通。

你也可以进一步检查服务器端监听情况：

```bash
ss -lnt | grep 10808
```

如果看到服务器本机监听了 `127.0.0.1:10808`，说明 `ssh -R` 已经建立成功。

## 四、示意图

```text
┌─────────────┐                              ┌─────────────┐
│  本地电脑    │                              │  云服务器    │
│             │                              │             │
│  代理软件    │                              │  使用代理    │
│  127.0.0.1  │  ←──── SSH隧道转发 ────→     │  127.0.0.1  │
│  :10808     │                              │  :10808     │
│             │                              │      ↓      │
└─────────────┘                              │  访问Google │
                                             └─────────────┘
```

从流量方向上理解就是：

1. 服务器访问 `127.0.0.1:10808`
2. SSH 把这个连接带回本地电脑
3. 本地代理软件代替服务器去访问目标网站
4. 结果再通过 SSH 隧道返回服务器

## 五、一键命令

如果你想在本地直接把 SSH 放到后台运行，可以执行：

```bash
# 本地执行，-N 表示不执行远程命令，只做端口转发
ssh -R 10808:127.0.0.1:10808 admin@你的服务器IP -N -f
```

`-f` 参数会让 SSH 在认证完成后切到后台运行。

更稳妥的写法可以加上失败即退出：

```bash
ssh -f -N -o ExitOnForwardFailure=yes -R 10808:127.0.0.1:10808 admin@你的服务器IP
```

如果你使用的是长期会话，建议进一步加上保活参数：

```bash
ssh -f -N \
  -o ServerAliveInterval=60 \
  -o ServerAliveCountMax=3 \
  -o ExitOnForwardFailure=yes \
  -R 10808:127.0.0.1:10808 \
  admin@你的服务器IP
```

## 六、为什么不能直接用公网 IP:端口

很多人一开始会尝试：

- 在本地代理软件上开启一个端口
- 然后让云服务器直接访问 `你的公网IP:代理端口`

这种方式经常失败，原因通常包括：

- 家宽网络没有公网 IP
- 运营商存在 NAT，公网无法直连你的电脑
- 本地防火墙阻止了入站连接
- 代理软件默认只监听 `127.0.0.1`
- 直接暴露代理端口本身存在安全风险

而 SSH 隧道转发的优势在于：

- 由本地主动发起 SSH 连接，不依赖公网入站访问
- 不要求服务器能直接打到你的家庭网络
- 不需要把代理端口直接暴露给公网

可以把两种方式概括为：

| 方式 | 能否成功 | 说明 |
|------|----------|------|
| 直接用公网 IP:端口 | 不稳定，常常失败 | 依赖公网入站、NAT、防火墙和端口暴露 |
| SSH 隧道转发 | 通常可行 | 由本地主动连出，更安全、更容易成功 |

## 七、常见问题

### 1. SSH 隧道建好了，但 `curl` 还是不通

优先排查以下几点：

- 本地代理端口是否真的可用
- `10808` 到底是 `HTTP` 还是 `SOCKS5`
- 服务器端环境变量是否设置正确
- SSH 会话是否已经中断
- 服务器是否把请求发到了错误的代理协议上

最常见的错误就是：

- 本地 `10808` 实际是 `SOCKS5`
- 服务器却配置成了 `http_proxy=http://127.0.0.1:10808`

这种情况下要改为：

```bash
export ALL_PROXY=socks5h://127.0.0.1:10808
```

### 2. 服务器上访问 `127.0.0.1:10808` 被拒绝

常见原因：

- 本地执行 `ssh -R` 的终端已经关闭
- SSH 登录失败但没有注意到报错
- 远程端口转发被服务器禁用了

可以在服务器上检查 SSH 配置：

```text
AllowTcpForwarding yes
```

如果服务器管理员禁用了端口转发，那么 `ssh -R` 无法正常工作。

### 3. 我想让服务器之外的其他机器也能访问这个代理

默认情况下，`ssh -R 10808:...` 通常只会让服务器本机访问 `127.0.0.1:10808`。这其实是更安全的默认行为。

如果你确实要让远程机器通过服务器公网 IP 访问这个端口，需要额外处理：

- 使用类似 `ssh -R 0.0.0.0:10808:127.0.0.1:10808 ...`
- 服务器端 `sshd_config` 可能需要开启 `GatewayPorts yes` 或 `GatewayPorts clientspecified`
- 同时还要考虑防火墙和安全风险

这已经不是“仅供服务器本机使用”的场景了，建议谨慎操作。

### 4. 隧道经常断开怎么办

可以配合 SSH 保活参数：

```bash
ssh -N \
  -o ServerAliveInterval=60 \
  -o ServerAliveCountMax=3 \
  -R 10808:127.0.0.1:10808 \
  admin@服务器IP
```

如果是长期使用，也可以配合 `autossh` 自动重连。

## 八、实用脚本示例

下面是一份适合在本地执行的简化脚本：

```bash
#!/bin/bash
# 文件名：ssh-proxy-tunnel.sh

REMOTE_USER="admin"
REMOTE_HOST="你的服务器IP"
REMOTE_PORT="10808"
LOCAL_HOST="127.0.0.1"
LOCAL_PORT="10808"

ssh -f -N \
  -o ExitOnForwardFailure=yes \
  -o ServerAliveInterval=60 \
  -o ServerAliveCountMax=3 \
  -R ${REMOTE_PORT}:${LOCAL_HOST}:${LOCAL_PORT} \
  ${REMOTE_USER}@${REMOTE_HOST}
```

建立成功后，在服务器上按实际代理类型使用：

```bash
# HTTP / mixed 代理
export http_proxy=http://127.0.0.1:10808
export https_proxy=http://127.0.0.1:10808

# 或 SOCKS5 代理
export ALL_PROXY=socks5h://127.0.0.1:10808
```

## 九、总结

把本地代理“带到”服务器上，最合适的思路不是直接开放公网代理端口，而是使用 SSH 隧道转发。

最核心的流程就是：

1. 本地确认代理可用
2. 本地执行 `ssh -R 10808:127.0.0.1:10808 admin@服务器IP -N`
3. 服务器上把程序代理指向 `127.0.0.1:10808`
4. 用 `curl` 或实际业务程序验证是否连通

如果你只记住一句话，可以记成：

> 直接暴露公网代理端口不安全也不稳定，使用 SSH 隧道才是更正确、更安全的做法。
