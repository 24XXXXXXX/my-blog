用 `spring.config.import` 把 `.env` 当作 properties 配置源导入，从而让 `@Value` / `@ConfigurationProperties` 直接读取。
<!--more-->

## 配置示例

```yaml
spring:
  application:
    name: local-test
  config:
    import: optional:file:.env[.properties]
```

## 这行配置解决了什么问题

加上 `[.properties]` 后，Spring Boot 会把 `.env` 当成标准的 properties 文件去解析并加入配置源，因此不再需要手动用 Dotenv 预加载。

## 原理简述（为什么要写成 .env[.properties]）

`spring.config.import: optional:file:.env` 对无扩展名的文件在部分场景下可能不稳定/不识别。

写成 `optional:file:.env[.properties]` 等价于告诉 Spring：按 properties 规则读取 `.env`（即 `key=value`），因此它会被纳入 Spring 的配置体系，`@Value` 或 `@ConfigurationProperties` 都能正常读取。

## 注意事项（容易踩坑）

- **.env 内容格式**：必须符合 `key=value`，不要写成 `export KEY=value` 这种 Shell 风格。
- **空格与引号**：建议写成 `FOO=bar` 或 `FOO="bar"`（需要引号就加，引号会作为值的一部分或按 properties 规则处理）。
- **缺文件不报错**：`optional:` 的含义是文件不存在也不报错；如果你希望“缺文件直接失败”，去掉 `optional:`。
- **优先级理解**：`config import` 只是把文件纳入配置源；最终以 Spring Boot 的属性优先级规则为准（例如命令行参数、环境变量通常优先级更高）。

## 如何验证是否生效

- 在 `.env` 中写入：

```properties
MY_APP_NAME=hello
```

- 代码中读取：

```java
@Value("${MY_APP_NAME}")
private String myAppName;
```

结论：在你当前写法下，删除 `LocalTestApplication` 里那段 Dotenv 代码也能正常读取 `.env`。
